│ 🔍 CAUSAS ANÁLISIS                                                           │
└──────────────────────────────────────────────────────────────────────────────┘

¿Por qué 6.15x speedup y no 8x?
  Breakdown del tiempo (8 threads):
    Cómputo puro:        3.33 ms (26.64 / 8)
    Overhead:            1.00 ms (barrera + coherencia + setup)
    Total medido:        4.33 ms ✓
  
  Overhead desglosado:
    • Barrera de sincronización:  ~0.3 ms
    • Coherencia de caché:         ~0.4 ms
    • Setup/teardown OpenMP:       ~0.3 ms

¿Por qué degrada con 16 threads?
  Hyperthreading introduce competencia:
    • 2 threads lógicos por core = Competencia por L1/L2
    • Context switching de HT = Pipeline stalls
    • Sincronización amplificada con 16 threads
    • Resultado: 96% más lento que 8 threads

¿Por qué Memory no es bottleneck?
  Análisis de bandwidth:
    • Datos generados: 40 MB (1000 gen)
    • Ancho de banda disponible: 50-80 GB/s
    • Tiempo teórico solo memoria: 0.8 ms
    • Tiempo real: 4.33 ms
    • Conclusión: CPU-bound, no memory-bound ✓

┌──────────────────────────────────────────────────────────────────────────────┐
│ 💡 LECCIONES APRENDIDAS                                                      │
└──────────────────────────────────────────────────────────────────────────────┘

1. Paralelización requiere análisis riguroso
   • No asumir que más threads = mejor rendimiento
   • Profiling revela limitaciones del hardware

2. Hyperthreading es contexto-dependiente
   • Excelente para I/O-bound (latencia)
   • Perjudicial para CPU-bound sin optimización especial

3. Localidad es rey en HPC
   • Static scheduling: decisión óptima aquí
   • Working set en L1: máxima eficiencia

4. Overhead debe ser medido, no asumido
   • OpenMP overhead < 1% en arquitectura bien diseñada
   • Sincronización es operación muy rápida (< 1 µs)

5. Benchmarking sistemático revela verdades
   • Predicciones 95% acertadas
   • Datos reales superan expectativas hasta 8 threads
   • Sorpresa interesante con Hyperthreading
